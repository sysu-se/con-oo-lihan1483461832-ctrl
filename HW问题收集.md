### 1. 什么是“标量 derived store”，它在这次作业里有什么作用？

1. **上下文：**  
   在 HW1.1 中，我一开始不太理解为什么老师说“可以保留少量标量 derived store，但不要再生成整板级的 view model”。我最开始的疑问是：既然已经有 `Sudoku / Game`，为什么还会需要 store？后来我才理解，领域对象和 Svelte 响应式不是一回事。

2. **解决手段：**  
   通过查 Svelte store / derived store 的资料，并结合自己修改 `src/node_modules/@sudoku/stores/grid.js` 的过程，我理解了：  
   - `Game / Sudoku` 是普通 JS 对象，负责业务逻辑  
   - Svelte 组件真正依赖的是可订阅的 store  
   - `derived store` 适合保存一些**由已有状态推导出来的简单标量或轻量结果**，例如 `gameWon`、`invalidCells.length === 0` 这类结果  
   - 但如果把整块棋盘状态再额外包一层复杂 view model，就容易和领域对象重复，边界变乱  

   所以我现在理解的“标量 derived store”作用是：**保留少量、简单、只用于界面展示的派生状态，但核心盘面和历史逻辑仍然由领域对象负责。**

### 2. 为什么直接修改对象内部字段或二维数组，Svelte 界面不一定会自动刷新？

1. **上下文：**  
   这是我在 HW1.1 中最核心的困惑之一。HW1 里我已经实现了 `Sudoku / Game`，但是一开始只是“对象能工作”，并没有真正接进 Svelte。后来接入时我发现：即使 `currentGame.guess(...)` 已经执行，界面有时也不会按预期更新。我最开始误以为“对象变了，界面就会自动变”。

2. **解决手段：**  
   通过阅读 Svelte store 机制和反复对照 `grid.js`、`Keyboard.svelte`、`Actions.svelte` 的调用链，我理解了：  
   - Svelte 不会自动追踪普通对象内部字段变化  
   - `currentGame.guess(...)` 只是改了一个普通 JS 对象的内部状态  
   - 真正让 UI 重新渲染的是 `store.set(...)` 或 `store.update(...)`  
   - 组件里写 `$store`，本质上是订阅了 store；只有 store 通知更新，界面才会刷新  

   所以我最后采用了 **store adapter** 思路：  
   - adapter 内部持有 `Game`  
   - 用户输入先进入 adapter，再调用 `Game.guess()` / `undo()` / `redo()`  
   - 最后由 adapter 把 `currentGame.getSudoku().getGrid()` 等状态重新 `set` 到 store  
   - 这样 Svelte 才能正确刷新 UI  

   这让我真正理解了：**领域对象负责业务，store 负责通知界面更新。**

### 3. 为什么这次作业需要 `Sudoku` 和 `Game` 两个对象，而不能只用一个对象？

1. **上下文：**  
   在 HW1 刚开始时，我一度觉得“盘面、输入、撤销、重做都写在一个对象里也能跑”，不太理解为什么老师强调至少拆成 `Sudoku / Game`。后来在实现 Undo/Redo、序列化、接入 Svelte 的过程中，我才意识到职责边界的重要性。

2. **解决手段：**  
   通过对照作业要求和自己的代码，我最后把两者分工理解清楚了：  
   - `Sudoku` 表示**一个盘面**，负责  
     - 保存 `grid`  
     - `guess(move)`  
     - `clone()`  
     - `toJSON()` / `toString()`  
     - 冲突检查 / 完成检查  
   - `Game` 表示**一局游戏会话**，负责  
     - 持有当前 `Sudoku`  
     - 管理 `undoStack` / `redoStack`  
     - `guess()` / `undo()` / `redo()`  
     - 对外作为 UI 的主要入口  

   这样拆分后，我能明显感觉到：  
   - `Sudoku` 只关心“当前局面是什么”  
   - `Game` 只关心“这一局是怎么演变到现在的”  

   这也让我更容易解释为什么 UI 接入时应该让 adapter 持有 `Game`，而不是直接让组件去改棋盘数组。

## 1. 高亮逻辑应该放在组件里，还是抽到 adapter / 领域层里？

### 当前理解

我目前知道它的功能是：高亮与当前选中格处于同一行、同一列或同一宫的格子。

但我还没有完全想清楚，这类逻辑究竟应该继续放在组件里，还是也应该进一步抽到 adapter / 领域层里。

### 目前的判断

我现在的理解是：

- 它更像一种视图层高亮规则
- 而不是数独本身的核心业务规则

所以，放在组件里是可以接受的。

因为即使没有这部分高亮，数独的核心功能仍然成立，例如：

- 棋盘数据本身
- 合法输入
- Undo / Redo
- 序列化
- 题目状态管理

这些才更接近领域层真正需要负责的内容。

### 还没有完全想透的地方

不过，我也意识到一个问题：

如果以后高亮逻辑继续变复杂，例如除了同行、同列、同宫之外，还要加入：

- 相同数字高亮
- 冲突格高亮
- 候选数联动高亮
- 不同模式下的高亮切换

那么它就可能不再只是“组件里顺手写一点判断”这么简单了。

这时候，更合理的方式可能是：

- 把它从组件中抽出来
- 形成更明确的 view-model / selector
- 让组件只负责“渲染结果”，而不是自己计算高亮规则

### 目前结论

所以我现在的结论还不是绝对的，而是一个阶段性判断：

- 当前规模下：放在组件里可以接受
- 如果逻辑继续增长：适合抽成更清晰的 adapter、selector 或 view-model

这个问题我现在还没有完全想透，但我已经大致形成了这样的方向判断。

## 2. candidates / notes 应该怎样和领域对象更自然地结合？

### 当前情况

目前项目里的候选数和 notes 模式，主要还是保留了旧逻辑。

例如：

- `candidates.js`
- `Keyboard.svelte`

里面仍然有较多现成处理方式。

这次作业为了控制改动范围，我主要把这些部分接到了新的 `Game / Sudoku` 上：

- 正式输入
- Undo / Redo
- 真实盘面状态

但是 `candidates` 还没有被彻底纳入新的领域模型。

### 我想到过的几种方向

对于 candidates / notes 后续应该怎么设计，我目前想过几种可能：

#### 方向一：继续作为独立 store

把 `candidates` 继续看成一种 UI 辅助状态，不放进 `Game` 领域对象里。

这样做的优点是：

- 改动小
- 容易和现有代码兼容
- 不会让领域对象一下子变得太重

但缺点也比较明显：

- Undo / Redo 时不一定能自然回滚 notes 状态
- 盘面状态和候选状态会分散在不同地方
- 长期来看一致性不够强

#### 方向二：把 candidates 一起放进 `Game`

也就是让 `Game` 不只管理“正式填入的值”，还同时管理：

- notes
- candidates
- 与它们相关的历史记录

这样做的好处是：

- Undo / Redo 可以一起回滚
- 游戏状态更完整
- `Game` 真正成为“整局状态”的统一入口

但问题是：

- `Game` 的复杂度会明显上升
- 需要重新设计状态结构
- 代码改动范围会更大

#### 方向三：单独引入 `NoteState / CandidateState` 对象

也可以不把所有内容直接塞进 `Game`，而是额外引入一个专门对象，例如：

- `NoteState`
- `CandidateState`

然后由 `Game` 组合这些对象，或者由 adapter 统一管理它们之间的协作。

这种方式的好处是：

- 结构更清晰
- 职责划分更自然
- 以后扩展 notes 规则时更容易维护

但它也意味着：

- 设计成本更高
- 对当前作业来说可能有些超出范围

### 目前的阶段性理解

我现在还没有完全确定哪一种最合理。

但如果只是从这次作业的范围看，我觉得：

- 把 `candidates` 暂时保留为独立 store，是可以接受的
- 因为它更像是“增强交互体验”的附加状态，而不是完成基本题目流程所必须的最小核心

不过如果继续扩展这个项目，我觉得这会是下一个需要认真设计的点。

### 当前倾向

我现在更倾向于这样理解：

- 短期作业实现：`candidates` 可以先保留为 UI / store 层状态
- 长期演进：应该考虑把它纳入更完整的状态模型，至少要和 Undo / Redo 的一致性问题一起考虑

所以，这部分我目前是“有几个可行方向，但还没有最终定案”。

## 3. game.js 这种流程控制模块和新的 Game 领域对象，长期来看应该如何分工？

### 当前的困惑

现在项目里除了我新增的 `src/domain/Game.js` 之外，原工程里还有一个：

`src/node_modules/@sudoku/game.js`

它主要负责这些流程控制功能：

- `startNew`
- `startCustom`
- `pause`
- `resume`

这让我有时会混淆：

- 一个叫 `Game`
- 另一个也在做“游戏流程控制”

那么长期来看，这两者的边界到底应该怎么定？

### 我目前的做法

目前我的做法是：

- 先保留原来的流程控制模块
- 核心盘面和历史逻辑放进领域对象 `Game`
- 通过 store adapter 桥接给 Svelte

这样做的好处是：

- 不需要一次性推翻原工程结构
- 能够在较小改动范围内完成作业要求
- 也能体现“UI 不直接依赖旧逻辑，而是通过领域对象和 adapter 间接工作”的设计方向

### 我现在理解到的分工方向

从长期设计上看，我现在大致把它们区分成两层：

#### 1. 领域对象 `Game`

更适合负责：

- 当前数独盘面
- 用户输入后的状态变化
- Undo / Redo
- 序列化 / 反序列化
- 与一局游戏本身直接相关的数据和规则

也就是说，它更像是“一局游戏的核心状态模型”。

#### 2. 旧的 `game.js` 流程控制模块

更适合负责：

- 开始新游戏
- 自定义开局
- 暂停 / 恢复
- 与应用生命周期、页面流程相关的操作

它更像是“应用层 / 流程层控制器”，而不是数独规则本身。

### 为什么会混淆

之所以会混淆，主要是因为它们名字都叫“Game”，但实际上处理的是不同层次的问题：

- 一个偏领域状态
- 一个偏应用流程

如果名字和边界没有明确表达出来，就容易在阅读代码时产生重复感和职责重叠感。

### 可能的长期优化方向

如果继续往下做，我觉得未来可以考虑这些方向：

- 保留两个层次，但重新命名
- 例如把旧的流程模块改成更像：
  - `GameController`
  - `SessionManager`
  - `GameFlowService`

这样会比两个都叫 `Game` 更清楚。

或者也可以进一步整理职责：

- 把旧 `game.js` 中真正属于领域状态的部分继续收拢到新的 `Game`
- 把流程控制保留下来作为更外层模块

### 目前结论

所以我现在的阶段性理解是：

- 核心盘面与历史状态：交给新的领域对象 `Game`
- 开始、暂停、恢复等流程控制：保留在更外层的流程模块中

这样分层，在目前作业范围内是合理的。

但从长期来看，我还不确定最优结构到底是：

- 完全保留两个层次
- 还是继续把旧 `game.js` 的职责收拢、改名，或者抽象得更清楚

所以这个问题目前对我来说，仍然属于：

部分理解了分层方向，但还没有完全设计到位。
